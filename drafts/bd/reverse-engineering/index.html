This guide will focus on reverse engineering in a context of Discord modifications, which means parts of it will be specific to webpack. However, some of the points discussed here can be applied to other web applications as well.

#Toolchains

In modern web development code often goes through a non-trivial toolchain before it lands in a user's browser in production. These toolchains may include:
  • Type checking (e.g. TypeScript, Flow)
  • Transpiling code (e.g. Babel, sucrase, TypeScript, CoffeeScript)
  • Bundling modules (e.g. webpack, rollup, esbuild, Parcel, Browserify)
  • Optimizations like "tree-shaking" (JavaScript slang for dead code elimination)
  • Minification (size optimizations to save bandwith)
The resulting code often looks very different from what any human would write and can be difficult to understand without experience.

#Bundling with webpack

When bundling production code with webpack, the modules are converted for webpack's runtime module system - no matter which module system the source code makes use of. For example let us take this very simple ES module with a single import and two exports:
1import {foo} from "./other-module";
2
3export default foo;
4
5export const bar = 2;
After being transformed by webpack & the rest of the toolchain, it could look roughly something like:
112345: (m, e, r) => {
2 "use strict";
3 Object.defineProperty(e, "__esModule", {
4 value: !0
5 });
6 var f = r(67890);
7 e.default = f;
8 e.bar = 2
9},
Besides some added boilerplate code and minification, we can see the ES5 and ES6 syntax has been removed completely. The import statement has been transformed into a simple call r(67890). In this case the r parameter is webpack's internally used require() function, which allows importing exports from other modules. The passed number is the internal ID of the imported module. We can find the ID of our module at the start, it is the 12345 key of our module function. It is important to note that these module IDs can not be expected to be stable across multiple versions of the codebase.
The export statements have been transformed into simple assignments to the e parameter. When evaluating a module, webpack will pass m = module, e = module.exports and r = require to the module function. After the module function has finished, webpack will take module.exports and return it to whatever other module was requiring ours. It will also store the exports in a cache in order to avoid having to evaluate the module again.

#Production code patterns

We have already seen one commonly used minification pattern in our example above. Understanding what these sometimes weird patterns are equivalent to is helpful for reverse engineering. Here is a table of some of the most commonly seen minification patterns:
DistSourceExplanation
!0trueConverting a falsy number using the not operator yields true.
!1falseConverting a truthy number using the not operator yields false.
void 0undefinedUsing the uncommon void operator yields undefined.
for (;;)while (true)Use of the uncommon endless for loop.
Besides minification there is some other weird patterns found in production code. The otherwise rarely seen comma operator may see more use. It evaluates all expressions and returns the result of the last expression.
One special case where it is used is function calls. You may stumble on cases where a function obj.foo is called as (0, obj.foo)(arg1, arg2) rather than obj.foo(arg1, arg2). This seems counterintuitive as the resulting code is longer, right? The reason for this is this, as in the actual language keyword. By doing (0, obj.foo) we are "stripping the context" from the function. Use of this within the function will now evaluate to the global Window rather than the obj it is defined on.

#Tips & tricks

I will conclude this guide with a handful of tips for reverse engineering. Some of these are specific to whatever technologies are used in the web application.

#1. Learn the dev tools

This is the most generic advice to give, but also important. The dev tools are your best friend for any kind of reverse engineering. Learning how to properly use them will improve your reverse engineering process a lot.

#2. Use domain-specific dev tools extensions

When reverse engineering for example a React application, you should also make use of the React Developer Tools extension. They will give you easier and extra insight into the application. A bonus tip specific to React: data for the component selected in dev tools is accessible via the $r global.

#3. Use breakpoints for arguments

You can put breakpoints on functions and then attempt to trigger a "regular" call to them in order to reverse engineer their arguments. The dev tools will show what arguments have been passed. This can be extended to investigating local variables within a specific function you are trying to understand.

#4. Use scopes for captured variables

In Chrome's dev tools function values display a [[FunctionLocation]] as well as a [[Scopes]] attribute. The [[Scopes]] attribute is a list of scopes sorrounding the functions and can help quickly figure out what value captured variables have.

#5. Also look at use cases

When reverse engineering a part, also look into how it is used in or interacts with other parts of the codebase. It may help understanding that part as well as building a broader understanding of sections of the application. (Of course this does not necessarily apply to libraries bundled in the application.)

#6. Aim to understand

Your utmost goal when reverse engineering usually should be understanding what is happening behind the scenes. You do not need to actually reverse engineer the source of every single little piece of code when you feel like you understand what its purpose is. However, be careful with attempting to just "make it work", skipping parts, trial and error etc. This may lead to unwanted side effects and more problems down the road.
As with most things, experience is key. While working with the production code of a given application, you get more and more familiar with the structure and how things are typically done. After some time, you may even build a mindset that enables you to fairly reliably predict the purpose and usage of parts of the codebase without having to reverse engineer them.
